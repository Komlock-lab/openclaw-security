# ACP sessions_spawn サンドボックス継承不備

## 調査サマリー

| 項目 | 内容 |
|------|------|
| 調査日 | 2026-03-12 |
| カテゴリ | sandbox-escape |
| 重要度 | High |
| 関連アドバイザリ | 4件（継承バイパス3件 + ACPX RCE 1件） |
| 影響バージョン | v2026.3.0以前 |
| 修正バージョン | v2026.3.1 ~ v2026.3.7 |
| OpenClawへの影響 | マルチエージェント環境でサンドボックスが無効化される |

## 概要

OpenClawのACP（Agent Communication Protocol）は、マルチエージェント環境でエージェント同士が通信・連携するためのプロトコルである。`sessions_spawn` は新しいエージェントセッションを生成するAPIだが、このプロセスで**サンドボックス設定の継承が正しく行われない**脆弱性が複数発見された。

これにより、サンドボックス内で動作しているエージェントが、サンドボックス外（ホスト環境）で新たなエージェントセッションを起動できてしまい、**サンドボックスによる隔離が完全に無効化される**。

### なぜ重要か

1. **マルチエージェント環境の普及**: OpenClawはAgent-to-Agent通信を主要機能として提供しており、ACP利用は増加傾向
2. **サンドボックスの前提崩壊**: サンドボックスが「有効」でも、ACP経由でバイパスされると保護が無意味になる
3. **攻撃チェーンの起点**: プロンプトインジェクション → ACP spawn → ホスト実行 という攻撃チェーンが成立

## 脆弱性の詳細

### GHSA-p7gr-f84w-hqg5: クロスエージェントスポーンの継承未強制

| 項目 | 内容 |
|------|------|
| 深刻度 | High |
| 修正バージョン | >= 2026.3.1 |
| カテゴリ | Sandbox Escape |

**問題**: `sessions_spawn` がクロスエージェントスポーン（あるエージェントが別のエージェントを起動）する際に、親エージェントのサンドボックス設定を子エージェントに継承しなかった。

**影響**: サンドボックス内のエージェントが、サンドボックス外で動作する子エージェントを生成可能。子エージェントはホスト環境のファイルシステム、ネットワーク、コマンド実行にフルアクセスできる。

### GHSA-474h-prjg-mmw3: ACP runtime指定での継承バイパス

| 項目 | 内容 |
|------|------|
| 深刻度 | High |
| 修正バージョン | >= 2026.3.2 |
| カテゴリ | Sandbox Escape |

**問題**: `sessions_spawn(runtime="acp")` を指定すると、サンドボックス継承チェックがスキップされ、ホスト環境でのACP初期化が許可された。

**影響**: `runtime="acp"` パラメータを明示的に指定するだけで、サンドボックス制限を完全にバイパスしてホストACP環境を初期化可能。

### GHSA-9q36-67vc-rrwg: サンドボックス化されたACPスポーンのホスト初期化

| 項目 | 内容 |
|------|------|
| 深刻度 | Medium |
| 修正バージョン | >= 2026.3.7 |
| カテゴリ | Sandbox Escape |

**問題**: サンドボックス化された `/acp` エンドポイントへのスポーンリクエストが、ホスト環境でACPセッションを初期化できてしまう。v2026.3.1-3.2の修正後も、別のコードパスで同様の問題が残存していた。

**影響**: 修正済みバージョンでも、異なるAPIパスを通じてサンドボックスバイパスが可能だった。

### GHSA-6f6j-wx9w-ff4j: ACPX Windowsラッパーのcwdインジェクション

| 項目 | 内容 |
|------|------|
| 深刻度 | High |
| 修正バージョン | >= 2026.3.1 |
| カテゴリ | RCE |

**問題**: ACPXのWindowsラッパーがシェルフォールバックを使用する際、特定のパスパターンでcwd（カレントワーキングディレクトリ）インジェクションが可能。

**影響**: Windows環境でのリモートコード実行（RCE）。

## 攻撃シナリオ

### シナリオ1: プロンプトインジェクション経由のサンドボックス脱出

```
1. 攻撃者がメッセージ経由でプロンプトインジェクションを実行
2. LLMがsessions_spawnを呼び出し（runtime="acp"指定）
3. ホスト環境で新エージェントセッションが起動
4. 新エージェントがホスト環境でコマンド実行（サンドボックス制限なし）
5. シークレット窃取、設定改ざん、永続化等の攻撃を実行
```

### シナリオ2: 悪意あるスキルによるサンドボックス脱出

```
1. 悪意あるスキルがClawHubからインストールされる
2. スキル内のコードがsessions_spawnを呼び出し
3. サンドボックス継承がバイパスされ、ホスト環境でエージェントが起動
4. ホスト環境でのファイルアクセス、ネットワークアクセスを獲得
```

### シナリオ3: ACP Provenanceを利用した検出

v2026.3.8で導入された **ACP Provenance** 機能を使うことで、ACPセッションの出所を追跡可能:

```
- セッショントレースID: ACPセッションの親子関係を追跡
- 入力の出所メタデータ: どのチャンネル/ユーザーからの入力かを記録
- 異常検出: ホスト環境からの予期しないACP初期化を検出
```

## 修正の経緯

| バージョン | 修正内容 |
|-----------|---------|
| v2026.3.1 | sessions_spawnにサンドボックス継承を強制（GHSA-p7gr）、ACPXラッパー修正（GHSA-6f6j） |
| v2026.3.2 | runtime="acp"指定時の継承バイパスを修正（GHSA-474h）、ACP dispatch デフォルト有効化 |
| v2026.3.7 | /acpエンドポイント経由のホスト初期化を修正（GHSA-9q36） |
| v2026.3.8 | ACP Provenance機能の追加（出所メタデータ + セッショントレースID） |

3回のリリースにわたって同種の問題が繰り返し修正されている点は、**ACP周りのセキュリティ設計に根本的な課題がある**ことを示唆している。

## 防御推奨事項

### 即時対応
1. **v2026.3.7以降へのアップデート**: 3件すべてのサンドボックス継承バイパスが修正済み
2. **サンドボックスモードの確認**: `sandbox.mode: all` が設定されていることを確認
3. **ACP dispatch設定の監査**: `openclaw config get acp.dispatch` でACP設定を確認

### 中期対応
4. **ACP Provenanceの活用**: v2026.3.8のProvenanceメタデータを使った異常検出の実装
5. **sessions_spawnの制限**: 必要最小限のエージェントスポーンのみを許可するポリシーの設定
6. **ランタイムチェックRC-019の実施**: sessions_spawnサンドボックス継承の定期確認

### 長期対応
7. **ACP通信の暗号化と認証強化**: エージェント間通信の完全性と機密性の確保
8. **最小権限の原則の適用**: 子エージェントは親エージェントと同等以下の権限のみを持つ設計
